Module 1 — Introduction to Agentic Coding

What Is an Agent?

An agent is an AI system that can observe, reason, and act in a loop until a goal is achieved.

1. Observe

Reads your codebase, open files, terminal output, and environment

2. Reason

Plans what to do based on the goal you gave it

3. Act

Writes code, runs commands, calls APIs, searches the web

Key insight: Unlike a one-shot chat response, an agent maintains a plan and keeps working until it succeeds or asks for clarification.

Module 2 — How LLMs Work

From Text to Tokens

LLMs don't read words — they read tokens. A token is roughly 3–4 characters of English text.

"GitHub Copilot is amazing!"
  ↓ tokenised ↓
["Git", "Hub", " Cop", "ilot", " is", " amazing", "!"]
  1      2      3       4      5      6             7   ← 7 tokens

Cost

Every token costs money and time to process

Limits

Fixed context window (e.g. 128K tokens). Older content gets dropped

Rule of Thumb

1,000 tokens ≈ 750 words ≈ ~40 lines of code

Module 2

The Context Window

Think of the context window as a sliding whiteboard — the model can only see what fits on it right now.

Context Window ┌───────────────┐ ┌─────────────┐ ┌─────────────────┐ │ System Prompt │ │ Chat Hist. │ │ Current Query │ │ (instructions) │ │ + files │ │ (your message) │ └───────────────┘ └─────────────┘ └─────────────────┘ ← Everything here influences the model's next token →

What Copilot automatically puts in your context:

  1. Currently open file
  2. Recently opened and related files
  3. Your instruction files (see Module 3)
  4. Conversation history
  5. Your current message
Module 2

How the Model Generates Text

LLMs are next-token predictors. Given all tokens so far, the model outputs a probability distribution and samples the next token.

// Input:  "def add(a, b):\n    return"
// Model:  P("a+b") = 72%, P("a + b") = 18%, P("sum(a,b)") = 6%, ...
// Output: "a + b"   ← sampled token

Temperature Controls Creativity

0.0 — Deterministic

Always picks the highest probability. Great for code generation.

0.7 — Balanced

Some creativity. Good for explanations and documentation.

1.5 — Creative

High randomness. Useful for brainstorming and ideation.

Module 2

Prompt Structure That Works

The model uses attention mechanisms to decide which earlier tokens matter when generating the next one.

Effective Prompt Template

[Role / persona]      ← who the model should act as
[Background context]  ← what it needs to know
[The task]            ← what you want it to do
[Constraints]         ← what it must or must not do
[Output format]       ← how you want the result
[Examples]            ← show a sample (optional)

Good Prompt

You are a senior TypeScript engineer.
This project uses Express 4 and Jest.
Refactor the getUserById function to:
- Use async/await
- Return null instead of throwing
- Keep existing JSDoc comments
Return only the refactored function.

Bad Prompt

Fix the user function.
Module 3 — Instruction Priority Order

The Four Places Copilot Reads Instructions

Priority (highest → lowest) ───────────────────────────────────────────────── 1. Inline prompt ← what you type right now 2. Agent definition file ← .github/agents/<name>.md 3. Repo instruction file ← .github/copilot-instructions.md 4. User-level instructions ← VS Code settings ─────────────────────────────────────────────────

Important: Higher priority does not erase lower priority. All sources are combined. When they conflict, the model follows the most recently / most prominently stated rule.

Module 3

How the Model Resolves Conflicts

Recency Bias

Later instructions (closer to the generation point) usually win

Specificity

More specific instructions override general ones

Repetition

If an instruction appears multiple times, it is reinforced

Conflicting Instructions

# copilot-instructions.md
Always use async/await for
asynchronous code.

# Agent file
Use Promises with .then()/.catch()
for all async operations.

Result

The agent file (higher priority) will likely win.

Keep instructions consistent across all files to avoid confusion.

Module 4 — Setting Up Copilot Instructions

The Most Impactful File in Your Repo

.github/copilot-instructions.md is automatically included in every Copilot session. Think of it as an onboarding doc for an AI engineer joining your team.

Project Overview

1-3 sentences: what this project is

Tech Stack

Runtime, framework, DB, testing, linting, CI

Conventions

TypeScript strict, naming, file structure, max function length

Testing Rules

Every function needs a test, 80% coverage, real DB in integration

Security Rules

No PII logging, parameterised queries, Zod validation

Output Format

Code only unless asked, JSDoc comments, TODO markers

Module 4

The CRAFT Framework

Use CRAFT to write instructions that are easy for the model to follow:

LetterStands ForQuestion to Ask
CClearWould a new engineer understand this without asking?
RRelevantIs this specific to this repo, not generic advice?
AActionableCan the model actually follow this instruction?
FFocusedDoes each instruction cover one thing only?
TTestableCan I verify the model is following this instruction?

Anti-patterns to avoid: "Write good code" (vague), "Be careful with security" (unactionable), "Use best practices" (undefined), instructions over 2000 tokens (crowds out code context).

Module 5 — Skills & Tools

What Is a Skill?

A skill (tool) is a capability an agent can invoke to interact with the world. Without skills, an agent can only read. With skills, it can act.

┌─────────────────────────────────┐ │ Agent │ │ ┌──────────────────────────┐ │ │ │ LLM Core │ │ │ │ (reasoning + language) │ │ │ └──────────┬───────────────┘ │ │ │ calls │ │ ┌──────────▼───────────────┐ │ │ │ Skills │ │ │ │ read_file write_file │ │ │ │ run_terminal web_search │ │ │ │ call_api create_pr │ │ │ └──────────────────────────┘ │ └─────────────────────────────────┘
Module 5

Built-In Copilot Skills

SkillWhat It Does
read_fileReads the contents of a file in the workspace
write_fileCreates or overwrites a file
list_directoryLists files in a directory
run_terminal_commandExecutes a shell command and captures output
search_codebaseSemantic search across all files
search_webSearches the web (where enabled)
create_pull_requestOpens a GitHub PR
create_issueCreates a GitHub Issue

Principle of Least Privilege: Grant only the skills the agent actually needs. A code reviewer needs read_file and search_codebase, not run_terminal_command.

Module 6 — Copilot CLI

AI in Your Terminal

Copilot CLI brings AI assistance directly to your terminal via the gh CLI.

gh copilot suggest

Converts natural language into shell commands.

gh copilot suggest "find all
TypeScript files modified in
the last 7 days"

# Output:
find . -name "*.ts" -mtime -7

gh copilot explain

Explains an existing command in plain language.

gh copilot explain "find .
-name '*.ts' -mtime -7
-not -path '*/node_modules/*'"

# Searches recursively for .ts
# files modified within 7 days,
# excluding node_modules

shell

General commands, file ops, pipelines

git

Branches, commits, rebases, worktrees

gh

PRs, issues, releases, workflows

Module 6

Practical CLI Patterns

Quick Command Lookup

gh copilot suggest "extract a
.tar.gz to a specific directory"

No more Googling tar flags!

Debug a Failing Command

gh copilot explain "git rebase
--onto main feature/old feature/new"

Understand before you fix.

Script Generation

gh copilot suggest "bash script
that reads repos from repos.txt
and clones each one"

Shell Integration

# Add to ~/.zshrc
copilot() {
  gh copilot suggest "$*"
}

# Then use anywhere:
copilot "delete merged branches"

Key Takeaways

LLMs Are Token Predictors

They predict the next token based on probabilities. Understanding this helps you write better prompts and manage context budgets.

Instructions Compound

Good instruction files improve every agent step. Use the CRAFT framework, keep them concise and conflict-free.

Least Privilege for Skills

Grant agents only the tools they need. A read-only analyst doesn't need run_terminal_command.

Specialise Your Agents

Create focused agent personas with clear roles. Coordinate multiple agents for complex tasks.

The big picture: You are not replaced by AI — you are amplified. The developer who understands how LLMs work and knows how to control them will ship faster, with fewer bugs, and with better documentation.
